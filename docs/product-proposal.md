# Product Proposal — FoodFlow

## 1. Обоснование идеи

### Прикладная задача

Современная семья сталкивается с несколькими взаимосвязанными проблемами в управлении питанием:

1. **Потери продуктов**: Домохозяйства выбрасывают 20–30% купленных продуктов. Основная причина — отсутствие системы отслеживания сроков годности и планирования меню.
2. **Когнитивная нагрузка**: Ежедневное планирование питания с учётом предпочтений, аллергий, целей по здоровью и наличия продуктов — сложная многофакторная задача, которую люди решают "в уме" или не решают вовсе.
3. **Разрозненность инструментов**: Люди используют разные приложения для рецептов, списков покупок и трекинга КБЖУ — нет единой системы, которая связывает инвентарь, план и покупки.
4. **Медицинские ограничения**: Аллергии, диабет, непереносимость лактозы или глютена требуют постоянного контроля при планировании меню — что трудно делать вручную для всей семьи.
5. **Учет предпочтений**: Меню рассчитанное на домохозяйство требует учета предпочтений всех жильцов, которые бывает сложно рассчитывать.

**FoodFlow** решает эти задачи через единый разговорный интерфейс с агентной системой, которая знает состав холодильника, профили членов семьи и умеет планировать питание с автоматическим учётом всех ограничений.

## 2. Цель и метрики

### Цель проекта

Создать PoC агентной системы, которая демонстрирует способность:
- автономно управлять инвентарём домохозяйства через natural language интерфейс
- генерировать валидные планы питания с учётом ограничений без участия пользователя в процессе валидации
- автоматически формировать список покупок на основе плана и текущего инвентаря

### Метрики успеха

#### Продуктовые метрики
| Метрика | Целевое значение |
|---|---|
| Точность распознавания продуктов из текста | > 90% (ручная проверка) |
| Доля планов, соответствующих диетическим ограничениям | 100% (нулевая толерантность к нарушению аллергий) |
| Доля планов с использованием продуктов с истекающим сроком | > 70% (если такие есть) |
| Точность автодобавления в shopping list | > 95% релевантных позиций |

#### Агентные метрики
| Метрика | Целевое значение |
|---|---|
| Доля корректных handoff-маршрутов (Router -> правильный агент) | > 85% |
| Доля планов, проходящих валидацию с первой попытки | > 60% |
| Среднее количество retry в цикле Plan→Validate→Refine | < 1.5 |
| Tool call accuracy (правильные аргументы у вызовов) | > 90% |

#### Технические метрики
| Метрика | Целевое значение |
|---|---|
| p95 latency ответа агента (текст) | < 8 сек |
| Uptime сервиса (FastAPI backend) | > 99% за время демо |
| Корректность записи в CSV (schema compliance) | 100% |

## 3. Потенциальные сценарии использования

### Базовые сценарии (MVP)

**SC-01: Пополнение инвентаря голосом**
- Пользователь: "Купил молоко 1 л, курицу 1 кг (срок до 8 марта), 10 яиц"
- Агент: парсит сообщение, вызывает `update_inventory` x3, подтверждает
- Результат: fridge.csv обновлён

**SC-02: Планирование меню на день**
- Пользователь: "Что поесть завтра?"
- MenuAgent: анализирует инвентарь, срок годности, профили семьи → генерирует 3 приёма пищи → ValidatorAgent проверяет → сохраняет
- Результат: meal_plans.csv + auto-shopping list

**SC-03: Многодневное планирование**
- Пользователь: "Спланируй еду на всю неделю"
- MenuAgent: MultiDayMealPlan (7 дней), валидация каждого дня, batch сохранение
- Edge case: если на 5-й день не проходит валидацию за 3 попытки — план для остальных дней сохраняется, пользователь предупреждён

**SC-04: Учёт приёма пищи**
- Пользователь: "Пообедал гречкой с куриным филе"
- RouterAgent: log_history, Open Food Facts → КБЖУ в nutrition_log, убирает ингредиенты из fridge
- Результат: history.csv + nutrition_log.csv

**SC-05: Управление блюдами ротации**
- Пользователь: "Добавь овсянку как регулярный завтрак" / "Куриный суп — наше воскресное блюдо"
- RouterAgent: ставит флаг `is_rotation=true` в dishes.csv; опционально сохраняет частоту ("каждый день", "раз в неделю") или привязку к дню недели
- Результат: MenuAgent предпочитает ротационные блюда при прочих равных — как безопасный выбор когда нет других сильных сигналов. Правило повторов из history.csv применяется в полную силу: если блюдо было недавно — не предлагать, если только пользователь явно не указал что ест его каждый день
- Edge case: все ротационные блюда требуют продукта которого нет → агент предупреждает и предлагает обычное блюдо

### Edge-кейсы

**EC-00: Холодный старт — нет информации о пользователе**
- Состояние: people.csv пуст, history.csv пуст, dishes.csv пуст, нет ротационных блюд
- Пользователь: "Спланируй питание на завтра"
- Ожидаемое поведение: агент генерирует план на основе только инвентаря, без персонализации; явно сообщает что рекомендации базовые и предлагает уточнить предпочтения ("Расскажи кто будет есть и есть ли ограничения — тогда план будет точнее")
- Не блокирует выполнение: план создаётся даже без профиля

**EC-01: Конфликт аллергий в плане**
- Инвентарь: арахисовое масло в холодильнике
- Профиль: сын аллергик на арахис, остальные без ограничений
- Логика: блюда могут быть назначены конкретному члену семьи — ValidatorAgent проверяет ограничения только того, кто это блюдо ест. Блюдо с арахисом допустимо если оно явно помечено "для взрослых", но недопустимо если назначено сыну или всей семье
- Ожидаемое поведение при плане на всю семью: агент разделяет блюда по едокам или предлагает вариант без арахиса для совместного ужина; не отклоняет план целиком

**EC-02: Пустой холодильник**
- Инвентарь: холодильник пуст, пантри тоже
- Пользователь: "Что приготовить?"
- Ожидаемое поведение: агент формирует план из базовых блюд, всё добавляет в shopping list, сообщает: "У вас нет продуктов. Вот план и список покупок."

**EC-03: Противоречивые ограничения**
- Профиль: веган + цель "набор мышечной массы" + нет бобовых
- Ожидаемое поведение: агент генерирует план с максимально доступными белковыми источниками (тофу, орехи), добавляет предупреждение об ограниченности вариантов

**EC-04: Срок годности завтра**
- Фрикаделька с истекающим сроком завтра
- Ожидаемое поведение: этот продукт ОБЯЗАТЕЛЬНО в плане на сегодня или завтра, невзирая на другие предпочтения

**EC-05: Нераспознанный запрос**
- Пользователь: "Помоги с заданием по математике"
- Ожидаемое поведение: RouterAgent вежливо сообщает, что система специализируется на питании и управлении кухней

**EC-06: Галлюцинация рецепта**
- ValidatorAgent должен отловить: агент предложил блюдо с "мясом тако" когда пользователь вегетарианец
- Ожидаемое поведение: отклонение + пересмотр

## 4. Ограничения

### Технические ограничения (SLO)

| Параметр | Значение | Обоснование |
|---|---|---|
| p95 latency (текстовый запрос) | < 12 сек | Reasoning-модель медленнее стандартных chat-моделей |
| Max retries в Plan→Validate | 3 | Баланс качества и стоимости токенов |
| Max items в shopping list (контекст) | 50 | Ограничение длины промпта |
| Concurrent users | 1 | PoC, один инстанс без session isolation |
| Max chat history (messages) | ~50 | Ограничение контекстного окна модели |

### Операционные ограничения

| Параметр | Значение | Примечание |
|---|---|---|
| Стоимость OpenAI API (PoC) | ~$10–50/мес | Зависит от выбора моделей и интенсивности использования |
| Стоимость одного планирования (7 дней) | ~$0.10–0.80 | Зависит от модели и количества retry |
| Хранилище данных | CSV файлы | Не масштабируется; для production нужен DB |
| Аутентификация | Отсутствует | PoC; в production нужен Auth layer |
| Резервное копирование данных | Ручное | Нет автоматических бэкапов в PoC |
| Поддерживаемые языки | EN, RU | Переключение через настройки; перевод через LLM |

## 6. Data Flow и распределение LLM/Agent vs детерминированный код

### Что делает LLM/Agent

| Задача | Компонент | Обоснование |
|---|---|---|
| Маршрутизация запроса | RouterAgent | Понимание намерения пользователя — NLU задача |
| Извлечение сущностей из текста | RouterAgent + ShoppingAgent | "купил молоко 1 л срок до 8 марта" → структурированные поля |
| Генерация плана питания | MenuAgent | Творческая задача с множеством ограничений |
| Валидация плана | ValidatorAgent | Семантическая проверка (не только regex) |
| Распознавание фото продуктов | RouterAgent (GPT-4o Vision) | Задача computer vision |
| Оценка КБЖУ блюда | RouterAgent | Приближённая оценка без точной БД нутриентов |

### Что делает детерминированный код

| Задача | Компонент | Обоснование |
|---|---|---|
| CRUD операции с CSV | DataManager | Точность данных, нет нужды в LLM |
| Сборка контекстного снимка | ContextAssembler | Форматирование данных для промпта |
| Схемная валидация tool call аргументов | Pydantic models | Типобезопасность |
| HTTP routing и CORS | FastAPI | Инфраструктура |
| Сортировка по сроку годности | ContextAssembler | Простая сортировка строк |
| Управление retry циклом | MenuAgent._generate_valid_plan | Инфраструктурная логика |

## 7. Потенциальные улучшения (за рамками PoC)

- **Barcode scanner**: Интеграция с Open Food Facts API для автоматического добавления продуктов по штрихкоду
- **Grocery delivery API**: Интеграция с Яндекс.Лавка или Сбермаркет для отправки shopping list
- **Nutritional database**: Замена LLM-оценки КБЖУ на верифицированную БД (Edamam, FatSecret API)
- **Calorie trend analytics**: Дашборд с графиком КБЖУ по дням/неделям
- **Meal cost estimation**: Оценка стоимости блюда на основе цен из магазина
- **Seasonal recommendations**: Учёт сезонности продуктов при планировании
- **Family calendar integration**: Учёт расписания (кто дома в какой день) при планировании порций
- **SQLite/PostgreSQL**: Замена CSV на реляционную БД для надёжности и concurrent access
